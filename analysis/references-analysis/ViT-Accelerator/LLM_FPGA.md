# LLM_FPGA (ViT-Accelerator/LLM_FPGA) 정밀 분석

> 분석 대상: `REF/ViT-Accelerator/LLM_FPGA/W8A8_version/{bert_12L, GPT_decoding_24L, LLaMA_decoding_32L, LLaMA_prefilling_32L}`
> 분석 방법: 실제 소스(`kernel.h`, `bert_region_{1,2,3}.cpp`, `host.cpp`, `kernel.cfg`, `Makefile`) Read 기반. 라인 근거 표기.

## 1. 개요

- **무엇인가**: Vitis HLS(C++)로 작성된 **W8A8(weight int8 / activation int8) 양자화 Transformer 인코더/디코더 레이어**를 AMD/Xilinx FPGA에 가속하는 커널 모음이다. 한 Transformer 레이어를 **3개의 HLS 커널(region_1/2/3)** 로 분할하고, 호스트(`host.cpp`)에서 레이어 수만큼 반복 호출(BERT 12회, GPT 24회, LLaMA 32회)하여 전체 모델을 수행한다.
- **한 줄 요약**: "int8×int16(=두 int8 weight 패킹) 시스톨릭 GEMM + int32 누산 + float 스케일 재양자화" 데이터패스를, QKV·Attention·FFN 3-region 데이터플로우 파이프라인으로 구성하고 HBM 멀티뱅크 + SLR 분할로 매핑한 Transformer 레이어 가속기.
- **코드 생성 출처(중요)**: `host.cpp:1-3` "Auto generated by Allo", `Makefile:19`의 경로 `.../project/transformer/heterocl_file/...` 로 보아 **HeteroCL/Allo(MLIR 기반 HLS DSL) 컴파일러가 생성한 HLS 코드**다. 커널 본문의 `// L19, L40, L41 ...` 주석은 DSL 원본 라인 번호 흔적이다. 따라서 BERT/GPT/LLaMA 세 모델 모두 동일 템플릿에서 dimension만 바꿔 생성된 구조다.
- **타깃 디바이스(코드 확인)**:
  - HLS 합성 리포트 헤더에 **`xcu280-fsvh2892-2L-e`(Alveo U280, virtexuplus)** 명시 (`GPT_decoding_24L/reports/.../Bert_layer_dataflow_region_2_csynth.rpt:11-12`), Vitis HLS **2022.1.2**.
  - `host.cpp:15-26`은 **HBM 32채널 + DDR 2채널**을 정의하고 실제로 HBM[0]~HBM[16]을 사용 → U280(8GB HBM2) 전제.
  - 단, `Makefile:26`의 기본 `PLATFORM ?= xilinx_u250_gen3x16...`는 **AMD 예제 makefile의 placeholder**(U250은 HBM 미탑재)로, 실제 빌드는 U280 플랫폼을 지정해야 함. → 본 repo의 실효 타깃은 **Alveo U280**으로 판단.
- **관련 연구 맥락**: 본 디렉토리 자체에는 README/논문 부재. 다만 상위 `REF/ViT-Accelerator/` 형제 폴더에 FlightLLM, EdgeLLM(`llama-fpga/au250EdgeLLM`), TATAA, ternaryLLM 등 LLM-FPGA 가속 레퍼런스가 존재. 본 구현은 그들과 동일 계열(HBM Alveo + HLS dataflow + INT8 GEMM)이나, 독자적으로 "int16에 int8 weight 2개 패킹"하는 DSP 패킹 기법을 사용한다는 점이 특징이다.

## 3. 핵심 모듈 정밀 분석 ★

### 3.0 왜 한 레이어를 3개 region으로 쪼갰는가
`LLaMA_decoding_32L/kernel.cfg:23-30`이 결정적 근거다:
```
slr=Bert_layer_dataflow_region_1_1:SLR0
slr=Bert_layer_dataflow_region_2_1:SLR1
slr=Bert_layer_dataflow_region_3_1:SLR2
stream_connect=...region_1_1.outp_k : ...region_2_1.outp_k : 16   (등 5개 스트림)
```
- **이유 (1) SLR(die) 분할**: U280은 3개 SLR(Super Logic Region)로 구성된다. region_1/2/3을 각각 SLR0/1/2에 배치하여 **단일 SLR 리소스 한계(특히 DSP/URAM)를 회피**하고 SLR간 배선 혼잡을 줄인다. 한 region이 단일 SLR에 들어갈 크기가 되도록 레이어를 3분할한 것.
- **이유 (2) region 간 free-running 스트리밍**: region 경계는 m_axi(HBM)가 아니라 **AXI-Stream(`#pragma HLS interface axis`)**로 직접 연결된다(`bert_region_1.cpp:281-284`의 outp_k/v/q/inp, `kernel.cfg`의 `stream_connect ... :16`(depth 16)). 즉 region_1의 출력 KVQ가 HBM 왕복 없이 region_2로 흘러가, **레이어 내부가 하나의 거대한 데이터플로우 파이프라인**이 된다.
- **이유 (3) dataflow 단위 분리**: 각 region top은 `#pragma HLS DATAFLOW`로 내부 함수들을 동시 실행(producer-consumer)한다. region을 나누면 dataflow 스코프가 작아져 HLS 스케줄링/검증이 가능해진다.

### 3.1 연산 코어 — int8×int16 패킹 시스톨릭 PE (★핵심 트릭)
`bert_region_1.cpp:13-35` `PE_int8_int16`:
```c
ap_int<8>  a = A_in.read();            // activation int8
ap_int<16> b = B_in.read();            // weight: int16에 int8 2개 패킹
ap_int<24> pack_b = (b(15,8) << 16) + b(7,0);   // 상위8b·하위8b를 24b 간격으로 배치
ap_int<16> c1, c0; (c1, c0) = a * pack_b;        // 단일 곱셈으로 a*w_hi, a*w_lo 동시 산출
c1 = c1 + c0[15];                                // c0 부호비트 보정(borrow)
C_out1 += c1;  C_out0 += c0;                      // 두 출력 채널 int32 누산
```
- **핵심 기법**: weight 두 개(int8)를 하나의 int16(상위/하위 바이트)로 패킹하고, activation int8과 곱할 때 **한 번의 곱셈(`a*pack_b`)으로 두 개의 부분곱(a·w_hi, a·w_lo)을 동시에** 얻는다. 이는 단일 DSP48E2에 두 MAC을 매핑하는 전형적 **DSP packing/double-pumping** 최적화로, INT8 GEMM의 DSP 효율을 2배로 끌어올린다(`kernel.h:39` `io_pack_int16 = ap_uint<8*w_parallel>`가 weight를 항상 짝수 패킹).
- **누산 폭**: 두 출력 채널 각각 `ap_int<32>`로 누산(line 18-19) → **int8×int8 곱(최대 ~16비트)을 inp_len(=768)회 누산해도 오버플로 없음**. 최종 64비트(`(C_out1, C_out0)`)로 묶어 drain.
- **시스톨릭 어레이**: `systolic_array_qkv`(`bert_region_1.cpp:37-94`)는 `block_size_a × block_size_b = inp_parallel(8) × (w_parallel/2)(8) = 8×8` PE 그리드. `A_fifo`/`B_fifo`는 SRL FIFO(`BIND_STORAGE ... impl=srl`), 각 PE는 `#pragma HLS UNROLL`로 공간 전개(line 68-74). 결과 누산기 `C[8][8]`는 `ARRAY_PARTITION complete dim=1,2`로 완전 분할(line 50-51) → 매 사이클 64개 MAC 동시.
- **데이터 흐름 3단계**: ① `data_load_AB`가 패킹 워드를 PE 행/열로 푸시 → ② `systolic_array` PE 격자가 II=1 누산 → ③ `data_drain_AB`(FIFO 비우기)+`data_drain_C`(64b 결과 출력). 전부 `#pragma HLS DATAFLOW`(line 53)로 겹친다.

### 3.2 GEMM 래퍼와 재양자화(requantize) — W8A8 파이프라인
`Linear_layer_qkv`(`bert_region_1.cpp:96-154`)가 한 타일 GEMM + bias + 스케일을 수행:
```c
ap_int<32> outp0_dp = acc.range(31,0) + bias[...];          // int32 + int16 bias
ap_int<8>  outp0    = outp0_dp * s[ps_offset + i];          // float scale → int8 재양자화(포화 cast)
```
- **재양자화 구조**: `int32 누산 → bias(int16) 가산 → per-row float scale s[] 곱 → int8 캐스팅`. 스케일 `s[seq_num]`는 **per-token(행) 동적 스케일**로 `const/buf17~28.h`에서 주입(미분석). 이는 SmoothQuant류 W8A8의 전형적 "activation per-token, weight per-channel" 양자화에 대응.
- **출력 패킹**: 두 결과(outp0/outp1)를 `double_pkt_int8`(int8 2배폭 AXI-Stream)로 묶어 next region에 전달(line 148-150). region_2/3의 모든 Linear(`Linear_layer_ds0/ds1/ds2`)도 동일 패턴이며, FFN(ds1/ds2)은 출력이 GELU/Res로 가야 하므로 int8이 아니라 **float(`converter_t`)** 로 dequant 출력(`bert_region_2.cpp:399-402`, `bert_region_3.cpp:210-213`).

### 3.3 입력 로더와 양자화(quantize)
- `input_loader_q/kv`(`bert_region_1.cpp:172-216`): HBM의 **float 입력**을 읽어 `data_temp.f *= s[]` 후 `ap_int<8>`로 캐스팅 → activation 양자화. `converter_t` union(`kernel.h:47-50`)으로 float↔uint32 비트 재해석.
- `input_loader`(line 156-170)는 원본 float을 **shortcut(residual)** 경로로 그대로 흘림(양자화 안 함). 즉 **residual은 FP32 정밀도 유지**, GEMM 경로만 INT8. (정확도 보존 설계)

### 3.4 Region 별 기능 정리 (bert_12L 기준)

| Region | 주요 sub-kernel (라인) | 역할 |
|---|---|---|
| **region_1** | `input_loader/_q/_kv`(156-216), `weight_loader_r1`(218-253), `systolic_array_qkv`(37), `Linear_layer_qkv`(96), top(263-321) | 입력 양자화 + **Q/K/V projection** 3개 GEMM. 출력 outp_k/v/q(int8) + outp_inp(float residual)을 AXI-Stream으로 region_2에 |
| **region_2** | `K_writer/V_writer/Q_buffer`(426-495), `systolic_array_attn/cont/ds0`(38/98/157), `Attention_layer`(216), `Softmax_layer`(498), `Context_layer`(284), `Linear_layer_ds0`(349), `Res_layer0`(551), `Layer_norm0`(591), top(650-710) | **Self-Attention 전체**: KV를 URAM에 저장(`BIND_STORAGE ... uram` line 666-668) → QK^T(attn) → softmax → ·V(context) → output projection(ds0) → residual0 → LayerNorm0 |
| **region_3** | `input_loader_ds1_res1`(284), `weight_loader_r3`(309), `systolic_array_ds1/ds2`(38/99), `Linear_layer_ds1/ds2`(160/222), `Gelu_layer`(343), `Res_layer1`(393), `Layer_norm1`(433), `output_writer`(481), top(504-559) | **FFN**: FC1(inp_len→gelu_len=3072) → GELU → FC2(3072→768) → residual1 → LayerNorm1 → HBM write |

#### Softmax (`bert_region_2.cpp:498-548`)
- 3단계: ① attn 결과(float)를 `buf[inp_parallel][seq_num]`에 적재 → ② `exp()` 누산으로 row-sum(`l_exp_sum`, line 528-535) → ③ `exp/sum` 정규화 후 **스케일 곱 → int8 재양자화**(line 540-542)로 context GEMM 입력 준비. 즉 softmax 출력도 INT8로 양자화하여 다음 GEMM에 투입. `buf`는 `ARRAY_PARTITION complete dim=1`(line 504)로 8-way 병렬. attention score는 region_2 line 270 `outp_dp / 8.0`으로 1/sqrt(head_len=64)=1/8 스케일 선반영.

#### LayerNorm (`bert_region_2.cpp:591-639`, `bert_region_3.cpp:433-479`)
- **표준 LayerNorm**: ① mean, E[x^2] 동시 누산(1-pass) → ② `var = E[x^2] - mean^2` (line 623) → ③ `(x-mean)/sqrt(var+1e-5) * gamma + beta` (line 630-631). gamma/beta는 `const/buf13~16.h`. `inp_parallel` 행 병렬, `sqrt`는 hls_math.

#### GELU (`bert_region_3.cpp:343-390`)
- **6-구간 piecewise-linear 근사**(line 373-384). `tanh`/`erf` 미사용 → LUT/연산 비용 절감. 출력 즉시 `* s[]`로 int8 재양자화(line 385). (주: line 373의 `if(inp<3) outp=0`은 첫 분기가 넓게 잡혀 음수 큰 영역을 0으로 보내는 형태로, 생성기 근사 테이블 그대로임 — 수치 검증은 미수행.)

#### Residual (`Res_layer0/1`)
- direct(GEMM 결과 float) + shortcut(보존된 FP32 입력)을 `converter_t`로 float 가산(`bert_region_2.cpp:583`). region_3의 `inp_res1` FIFO는 FFN 지연을 흡수하기 위해 **depth=32*768, URAM**(`bert_region_3.cpp:521-523`)으로 크게 잡음.

### 3.5 메모리(HBM 뱅킹 / array_partition / URAM)
- **HBM 뱅크 분리**: 각 region top의 m_axi 포트는 서로 다른 `bundle=gmem0..5`(`bert_region_1.cpp:275-280`)로 분리되고, `host.cpp:104-145` / `kernel.cfg:2-21`에서 **물리 HBM 채널에 1:1 배치**(in=HBM0, wk=HBM1, wv=HBM2, wq=HBM3, w_ds0=HBM4...). LLaMA는 가중치가 커서 wk/wv/wq/ds0/ds1/ds2를 **각각 2~4채널로 분산**(`LLaMA.../kernel.cfg`·`host.cpp:130-198`, HBM[1]~HBM[16])하여 대역폭을 합산.
- **array_partition(가중치)**: top에서 상수 weight/scale 버퍼를 `cyclic factor=2`(스케일/bias) 또는 `factor=8`(가중치 행, inp_parallel=8과 매칭)으로 분할(`bert_region_1.cpp:285-291`) → 한 사이클에 다수 weight 동시 read.
- **URAM 활용**: K/V 캐시(`bert_region_2.cpp:665-668` `ram_2p impl=uram`), 큰 중간 FIFO(softmax `sfm_outp` depth=512, gelu `outp_gelu` depth=3072, residual depth=24576)를 `impl=uram`로 지정 → BRAM 절약 + 지연 버퍼링.
- **SRL FIFO**: PE 간/짧은 스트림은 `impl=srl`(LUT 기반 시프트레지스터)로 BRAM 비소모.

### 3.6 BERT vs GPT vs LLaMA 차이

| 항목 | bert_12L | GPT_decoding_24L | LLaMA_*_32L |
|---|---|---|---|
| dimension(`kernel.h`) | seq512, inp768, head12×64, gelu3072 | (소스부재, 리포트상 BERT류 구조) | **inp4096, head32×128, gelu11008** (LLaMA-7B) |
| 병렬화 파라미터 | `inp_parallel=8, w_parallel=16` | - | **`w_parallel=64, block_size=32, head_parallel=4, block_num_ds{0,1,2}`** (`LLaMA kernel.h:19-25`) |
| Attention 구조 | Attention→Softmax→Context를 region_2 top에서 순차 연결(`bert_region_2.cpp:704-706`) | `head_spliter`/`Self_attention`/`head_merger` 융합 모듈(csynth 리포트 모듈명) | **`Self_attention` 래퍼가 head_parallel(4)개 head를 UNROLL하여 Attn+Softmax+Context 융합**(`LLaMA bert_region_2.cpp:374-400`) |
| Norm | LayerNorm (mean/var/γ/β) | LayerNorm | **여전히 LayerNorm** (RMSNorm 아님 — `LLaMA bert_region_3.cpp:332-352`) |
| 활성화 | GELU(6구간 PWL) | GELU류 | **여전히 GELU(동일 6구간 PWL)** (SiLU/SwiGLU 아님 — `LLaMA bert_region_3.cpp:277-303`) |
| 레이어 반복 | host에서 12회 | 24회 | 32회 |

- **(중요·정직성)** "LLaMA" 폴더는 **이름과 dimension만 LLaMA-7B**일 뿐, **RMSNorm·RoPE·SwiGLU(SiLU gate) 같은 LLaMA 고유 연산은 구현돼 있지 않다**. region_3는 BERT와 동일한 표준 LayerNorm + 6-구간 GELU를 그대로 사용한다(라인 근거: `LLaMA bert_region_3.cpp:277, 332, 351`). grep 결과 `rms/rope/silu/rotary/sigmoid` 키워드 매칭 0건. 즉 **BERT 데이터패스 위에 LLaMA 크기 워크로드를 얹은 성능 추정용 골격**으로 보는 것이 정확하다.
- **(중요·정직성)** "decoding" vs "prefilling" 폴더 차이도 소스상 거의 없다(`LLaMA_prefilling_32L/bert_region_1.cpp`의 PE/structure가 decoding과 동일). 둘 다 seq_num=512 전체 시퀀스를 GEMM으로 처리하며, **autoregressive 토큰별 디코딩(매 step KV cache 증분, seq_len=1 GEMV)은 구현돼 있지 않다.** KV는 region_2에서 URAM에 1회 적재될 뿐 step간 누적 캐시가 아니다. → "decoding/prefilling"은 폴더 라벨 수준의 구분.

## 4. 데이터플로우 / 실행 흐름

### 4.1 한 레이어 파이프라인 (region 간)
```
HBM(float inp, int8 weights)
  │ (m_axi, bundle gmem0..)
  ▼
[region_1 / SLR0]  input quant + QKV GEMM
  │ outp_k, outp_v, outp_q (double_pkt_int8) + outp_inp (pkt_float residual)
  │ ──── AXI-Stream (depth 16, HBM 우회) ────►
  ▼
[region_2 / SLR1]  KV→URAM, QK^T, Softmax, ·V, output-proj, +Res0, LayerNorm0
  │ outp_ln0 (pkt_float)  ──── AXI-Stream ────►
  ▼
[region_3 / SLR2]  FC1→GELU→FC2, +Res1, LayerNorm1
  │
  ▼ (m_axi outp_addr → HBM)
HBM(float output)  → 다음 레이어의 입력으로 host가 재투입
```
- **레이어 내부**: 완전 스트리밍(region 경계는 stream, region 내부는 `#pragma HLS DATAFLOW`로 producer-consumer 동시 실행). region_2 top(`bert_region_2.cpp:699-709`)은 9개 sub-kernel을 dataflow로 연결.
- **레이어 간**: `host.cpp:204-214` 루프가 region_1→2→3을 enqueueTask하고 `q.finish()`로 동기화 후 **12번 반복**. 가중치만 레이어마다 다르고 동일 xclbin/커널을 재사용(weight는 본 데모에서 동일 버퍼 재사용 — 단일 레이어 성능 측정용 데모).

### 4.2 양자화 파이프라인 (한 GEMM 기준)
```
float act ──(×s_act, cast)──► int8 act
int8 weight(×2 packed in int16) ──┐
                                  ▼
              systolic MAC: Σ int8·int8 → int32 (×2 채널, 64b)
                                  │ + bias(int16)
                                  │ × s_out(float, per-token)
                                  ▼
                    int8 (다음 GEMM) 또는 float (GELU/Res/Norm 경로)
```

### 4.3 병렬화 계층
1. **PE 공간 병렬**: bert 8×8=64 MAC/사이클, 각 MAC은 packing으로 2배 → 실효 128 int8-MAC/cycle/region(GEMM 코어). LLaMA는 `block_size=32`·`head_parallel=4`로 더 넓음.
2. **Pipeline II=1**: 모든 inner loop `#pragma HLS PIPELINE II=1`.
3. **Dataflow**: region 내 함수 동시, region 간 stream 동시(3-stage 매크로 파이프).
4. **Head 병렬(LLaMA)**: `Self_attention`이 head_parallel=4 head를 UNROLL 동시(`LLaMA bert_region_2.cpp:393-398`).

### 4.4 HBM 메모리 계층
- L0: PE 누산기(레지스터, `ARRAY_PARTITION complete`).
- L1: 스트림 FIFO(SRL/URAM), tile 버퍼 `A[]`(BRAM/URAM).
- L2: K/V 캐시(URAM, on-chip).
- L3: HBM(가중치/입출력, 채널 분산으로 대역폭 합산).

## 8. 강점 / 한계 / 리스크

**강점**
- **DSP 패킹(int8 weight 2개를 int16에)** 으로 단일 곱셈에 2-MAC → INT8 GEMM DSP 효율 2배 (`PE_int8_int16`).
- **3-region × SLR 분할 + AXI-Stream 체이닝**: HBM 왕복 없이 레이어 내부 완전 스트리밍, SLR 혼잡 완화.
- **혼합 정밀도**: GEMM은 INT8, residual/softmax/norm 누적은 FP32 유지 → 정확도 손실 최소화 설계.
- **메모리 자원 의식적 매핑**: URAM(KV/대형 FIFO) vs SRL(짧은 FIFO) vs BRAM 구분, array_partition factor를 병렬도와 일치.
- **확장성**: 동일 템플릿으로 BERT/GPT/LLaMA 차원 스케일(코드 생성기 기반).

**한계**
- **GPT는 소스 부재**: `GPT_decoding_24L`에 host/region cpp 없음 → 빌드 산출물·리포트만으로 추정 가능.
- **LLaMA가 LLaMA 아님**: RMSNorm/RoPE/SwiGLU 미구현, LayerNorm+GELU 그대로 → 정확도 측면에서 실제 LLaMA 추론 불가, **성능 골격(GEMM dimension)만 유효**.
- **decoding/prefilling 미구분**: 진짜 autoregressive KV-cache 증분 디코딩(GEMV, seq=1) 없음 → 디코딩 효율(메모리 바운드 특성) 미반영.
- **정확도 검증 부재**: host에 골든 비교 없음(`TEST PASSED`는 단순 통과). GELU PWL 근사(line 373 분기 포함)의 수치 정확도 미확인.
- **데모 수준 host**: 12레이어 루프가 동일 가중치 버퍼 재사용 → end-to-end 추론 파이프라인이 아닌 latency 측정용.
- **단일 레이어 인스턴스**: 한 번에 1 레이어만 온칩, 레이어 간 HBM 왕복(out→in 재투입) 비용 존재.

**리스크(재사용 시)**
- 코드 생성기(Allo) 산출물이라 **수작업 수정 시 일관성 깨지기 쉬움**(매크로/라인주석 의존).
- `kernel.cfg`의 HBM/SLR 바인딩이 특정 U280 셸에 종속 → 다른 디바이스 포팅 시 전면 재매핑 필요.
- packing PE의 `c1 = c1 + c0[15]` 부호 보정 로직은 **2의 보수/패킹 가정에 강결합** → weight 패킹 포맷 변경 시 버그 위험.

## 9. 우리 프로젝트(고처리량 ViT/Transformer FPGA 가속기 HG-PIPE 계열 + XR 시선추적) 관점 시사점

우리 타깃(HG-PIPE류 고처리량 ViT 가속 + XR eye-tracking, 저지연·고FPS)에서 **직접 재사용 가능한 구체 자산**:

1. **W8A8 packed-int8 시스톨릭 GEMM (`PE_int8_int16`)**: ViT의 patch-embed/QKV/MLP GEMM에 그대로 이식 가능. 단일 DSP 2-MAC packing은 HG-PIPE의 PE 밀도/throughput을 높이는 즉효 기법. → **우리 PE에 weight 2-packing + int32 dual-accum 패턴 채택 검토.**
2. **재양자화 블록(int32+bias→×float scale→int8)**: per-token activation scale + per-channel weight scale 구조(`Linear_layer_*`)는 ViT W8A8에 동일 적용. requant를 GEMM drain 단계에 융합(`l_bias_scale` 루프)하는 방식은 별도 재양자화 패스 제거에 유용.
3. **3-region × SLR 분할 + AXI-Stream 체이닝 (`kernel.cfg`)**: ViT 인코더 블록을 (attention / MLP / norm) 단위로 SLR에 나눠 배치 + stream_connect로 HBM 우회 → **HG-PIPE의 layer-pipelined 고처리량 구조와 정확히 부합.** 우리 멀티-SLR 매핑 템플릿으로 차용.
4. **HBM 멀티뱅킹(`host.cpp`+`kernel.cfg` sp=)**: 가중치를 채널 분산(LLaMA의 4채널 분산)해 대역폭 합산 → ViT의 큰 MLP 가중치 로딩 병목 완화. **포트별 bundle 분리 + HBM 채널 1:1 고정 패턴**을 그대로 적용.
5. **메모리 자원 정책**: KV/대형 reorder 버퍼=URAM, 짧은 FIFO=SRL, array_partition factor=병렬도 일치 → XR 실시간 경로의 BRAM 부족 회피에 직접 적용. residual을 FP32로 우회 보존하는 정밀도 절충도 시선추정 정확도 유지에 참고.
6. **Softmax/LayerNorm 스트리밍 구현**: 1-pass mean/E[x²] LayerNorm, exp-sum-정규화 후 int8 재양자화 softmax는 ViT attention에 재사용 가능(우리는 추가로 RMSNorm/GELU-tanh 정밀 버전 선택 가능).
7. **반면교사**: (a) "LLaMA가 LLaMA 연산을 안 함"처럼 **dimension만 바꾼 골격은 정확도 검증 없이는 무의미** → 우리 ViT 포팅 시 반드시 골든 대조 추가. (b) decoding(GEMV/메모리바운드) 미구현 → XR의 저배치·저지연(=GEMV 유사) 시나리오에선 본 repo의 GEMM-bound 구조를 그대로 쓰면 비효율, **저지연용 별도 데이터패스 필요.**

> 요컨대: **PE packing + requant fusion + 3-region/SLR stream 체이닝 + HBM 뱅킹**의 4개 패턴이 HG-PIPE류 ViT 가속에 가장 가치 있는 차용 대상이며, LLaMA/디코딩 부분은 "구조 참고용"으로만 취급.
